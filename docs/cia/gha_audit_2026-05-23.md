## 2. R-12 — Fork PR safety

**Risk:** If a PR from an attacker fork can trigger a workflow with secrets access, the attacker could exfiltrate secrets via `echo $SECRET >> file && git push`.

| Workflow | PR-triggered? | Secret access? | Verdict |
|---|---|---|---|
| `ci.yml` | ✅ pull_request | none used (only `node --check`) | ✅ safe |
| `handoff_lint.yml` (lint job) | ✅ pull_request | GITHUB_TOKEN read-only | ✅ safe (no write surface) |
| `handoff_lint.yml` (risk-register job) | ❌ push only (`if: github.event_name == 'push'`) | GITHUB_TOKEN write | ✅ never runs on PR |
| `julian_sync.yml` | ❌ schedule/manual only | CF_API_TOKEN, CF_ACCOUNT_ID | ✅ never runs on PR |
| `kb_extract.yml` | ❌ workflow_dispatch only | GROQ_API_KEY | ✅ requires manual launch by maintainer |
| `kb_extract_test.yml` | ❌ workflow_dispatch only | GROQ_API_KEY | ✅ |
| `main_sync_check.yml` | ❌ schedule/manual only | GITHUB_TOKEN | ✅ |
| `stale_branch_cleanup.yml` | ❌ schedule/manual only | GITHUB_TOKEN | ✅ |

**Critical**: No `pull_request_target` usage anywhere → forks cannot run with write access to repo or read access to secrets. ✅

**Verdict:** R-12 status = **mitigated by current trigger design. No changes needed.**

## 3. Action pinning audit (R-10 supply chain)

| Action | Pin version | Recommendation |
|---|---|---|
| `actions/checkout` | @v4 (ci, julian_sync, kb_extract*), @v5 (handoff_lint, main_sync, stale_cleanup) | unify to @v5 (latest LTS, no breaking change) |
| `actions/setup-node` | @v4, @v5 (mixed) | unify to @v5 |
| `actions/github-script` | @v8 | ✅ current |

**SHA pinning:** None of the actions are SHA-pinned. All are major-version tag pinned. Trade-off:
- Major-tag = auto-receive security patches from publisher (good)
- Major-tag = trust publisher (`actions/*` is first-party, low risk)
- SHA-pin = no automatic updates → Dependabot maintenance burden

**Recommendation:** Accept major-tag pinning for `actions/*` (first-party). SHA-pin only if we adopt third-party actions in future.

**Sub-recommendation:** **unify all to @v5 in Phase 1 P1-A** (5-min change, eliminates mixed-version confusion).

## 8. Other observations

### 8.1 `julian_sync.yml` uses `git pull --rebase` on automated commit
- Rebases against `origin/main` to avoid push race. Safe.
- If conflict → rebase fails, push fails, next run retries. ✅ safe failure mode.

### 8.2 `julian_sync.yml` writes to `data/julian_all.json` (15MB)
- Every 6 hours = 4 commits/day = 1460 commits/year of 15MB diff. **Repo bloat over time.**
- Not a security issue, but a **performance + repo health** concern. (Cross-link: P0-E perf baseline notes this.)
- Future: consider Git LFS or moving the canonical store to GitHub Release (already uploaded as `julian-data` release asset — could remove `data/julian_all.json` from repo). Decision for HORATAD/JULIAN session.

### 8.3 `bump_version.ps1` + `push.bat` in repo root
- Windows-only dev scripts. Not security-sensitive (no secrets).
- Could be moved to `scripts/` for cleanliness. Not GUARD's call.
