## WHY LOG (v4)

- **เลือก audit FB pipeline เป็นงานแรก** — infra ชุดใหญ่ที่สุดที่เพิ่มมาตั้งแต่ GUARD handoff ล่าสุด (5 workflow + 7 worker) ถือ token แรง (post/edit/delete Page) + มี destructive workflow → ตรง trigger SOP-03. GUARD เป็น owner ของ `.github/workflows/*` แต่ PLATFORM สร้างไว้โดยยังไม่ผ่าน security review → gap ที่ต้องปิด

- **R-20 = controls active ไม่ใช่ open** — token never-expire ฟังดูน่ากลัว แต่อยู่ใน GitHub Secrets เข้ารหัส + repo single-user + GHA mask อัตโนมัติ → P(รั่ว) ต่ำ. ความเสี่ยงจริงคือ "ถ้ารั่วแล้วคุมต่อเนื่อง" → แก้ด้วย rotation discipline (rotate ทุกไตรมาสแม้ไม่หมดอายุ) ไม่ใช่เปลี่ยน architecture

- **post_manage `contents:read` ดีกว่าปล่อย default** — workflow แค่เรียก Graph API ไม่ push git → ไม่ต้องการ write. ลด blast radius ถ้าวันหน้า token/dispatch ถูก abuse (ทำได้แค่อ่าน repo ไม่แก้). cost = 2 บรรทัด, benefit ชัด → ADOPT

- **backfill `!cancelled()` แทน `== 'success'`** — JULIAN รายงานว่า genealogy/accuracy backfill ไม่เคยรันเพราะ Job 1 timeout 55 นาที (result=failure) → Job 2 skip ทุกรอบ. backfill เป็น `--resume` idempotent + แยก data-repo → ปลอดภัยรันแม้ sync ไม่จบ. ใช้ `!cancelled()` ไม่ใช่ `always()` เพื่อให้ยังเคารพการ cancel ด้วยมือ (best practice GHA conditional)

- **R-22 ส่ง PLATFORM ไม่แก้เอง** — fb_autopost/fb_kpi/fb_post_manage = worker code (PLATFORM owns ตาม File Ownership). GUARD แก้ได้แค่ workflow yaml. cross-project rule → note ใน PLATFORM handoff ให้เปิด session PLATFORM ทำ (P3 ไม่เร่ง)
