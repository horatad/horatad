#### Q1: Julian day table format
- ไม่มี "per-day table" — เก็บ per-person โดย `jd` field (epoch Jan 1 2000 = 730428)
- `get_j(d,m,y)` = `730428 + days_since_jan1_2000` — ใช้ตรงกับ engine.js
- ไม่ใช่ astronomical Julian Day (JD 2451545 = Jan 1 2000) — เป็น custom epoch

#### Q2: Planet position granularity
- **sign-level (0-11)** = เพียงพอสำหรับ quality detection (มหาอุจ/เกษตร/นิจ/ฯลฯ ขึ้นกับ sign)
- degree-level available ถ้าต้องการ (raw_value / 60 = degree ใน 360°)
- Lagna = ต้องการ birth time → ไม่ include (ส่วนใหญ่ไม่มี time_utc ที่ verified)
- Moon error estimate: ±0.5 sign ถ้า actual birth time ≠ 06:00 (Moon เคลื่อน ~0.5 sign/วัน)

#### Q3: Index schema
- **ทำแล้ว** — `data/julian_natal_stats.json` (6.3KB) มี aggregate counts per quality per planet
- **ไม่ต้องมี index แยก** — ใช้ `data/julian_positions_by_jd.json` + join กับ `julian_all.json` แทน
- Query "คนทุกคนที่มี อาทิตย์อุจ":
  ```js
  const byJd = require('./data/julian_positions_by_jd.json');
  const records = require('./data/julian_all.json');
  const SU_IDX = 0; // planet_order[0] = SU
  const UC_SIGN = 3; // กรกฎ = อาทิตย์มหาอุจ (EXALT_MAP[1]=3)

  // → ~6,347 records in ~1ms
  ```

**How to use จาก BIBLE:**
```js
import { readFileSync } from 'fs';
const byJd = JSON.parse(readFileSync('data/julian_positions_by_jd.json'));
const records = JSON.parse(readFileSync('data/julian_all.json'));
const PLANETS = byJd.meta.planet_order; // [SU,MO,MA,ME,JU,VE,SA,RA,KE,MR]

// หา records ที่ อาทิตย์มหาอุจ (sign=กรกฎ=3)
const su_uc = records.filter(r => byJd.positions[r.jd]?.[0] === 3);

// หา records ที่ match หลายเงื่อนไข (เช่น SU=อุจ + JU=เกษตร)
const dual = records.filter(r => {
  const pos = byJd.positions[r.jd];
  return pos && pos[0] === 3 && pos[4] === 3; // SU=กรกฎ, JU=เมษ
});
```
