## WHY LOG
- **KB extract ทำได้ก่อน CSP enforce** — เพราะ KB extract เป็น data move ไม่ใช่ code structure change → ไม่กระทบ inline event handler scope ของ CSP
- **เลือก as-is extract แทน merge กับ v3/kb.json** — KB_RULES inline (V2.0.0) ใช้ schema ต่าง v3/kb.json (V2.3.0) แม้ rule count เท่ากัน (342) → safer to preserve current behavior. Merge ทั้งคู่ไว้รอบหน้าถ้าจะรวม legacy → V3 path
- **Async load + guard pattern** — ทั้ง `_matchRules` + `openInterpretation` มี `if(!_kbRules)return` guard อยู่แล้ว → KB load delay ไม่ทำให้ app crash, แค่ทำให้ปุ่ม ทำนาย ไม่ตอบสนองจนกว่า KB load เสร็จ
- **Net wire bytes ~unchanged** — KB เปลี่ยนจาก inline JS (198KB) เป็น JSON (194KB) ลดเล็กน้อย แต่ win หลักคือ parser time + cache isolation ไม่ใช่ wire savings
- **kb_embedded.json แทน kb.json** — ตั้งชื่อแยกเพราะ v3/kb.json (V2.3.0) เป็น BIBLE current ที่ใช้โดย v3/v3tab.js — ห้ามชนชื่อกัน
- **Step 0b (PROVINCES/astro tables) ไม่ทำ** — assessment: ใช้ใน `_core()` calc path (l.572,585) ที่ run ตั้งแต่ init → fetch async ทำให้ init พัง. ทางเลือก `<script src>` ไม่ลด bundle (แค่ขยับ HTTP request) → value/effort ต่ำ. รวมเข้า Step 1 ดีกว่า เมื่อจัด proper module
- **UX toast แทน silent return** — เห็นช่องว่าง UX: user กด ทำนาย เร็วกว่า KB load (race) → return เงียบไม่บอกอะไร. แก้ 1 บรรทัดด้วย `_showToast('กำลังโหลด KB... ลองอีกครั้ง')` — friendly + ไม่ตื่นเต้น
- **Cleanup leftover files** — `push.bat1`/`push.txt`/`test.txt` เป็นเศษ Windows workflow เก่าที่ถูกแทนด้วย `scripts/admin/*` แล้ว — ลบเพื่อ repo hygiene
- **หยุดที่ Step 0** — assess แล้วงานต่อไปทุกอย่างต้อง user input (CSP, Lighthouse, mobile test) หรือ design decision (julian_all.json target) → stop autonomously ดีกว่าทำงานที่ผูก premature
