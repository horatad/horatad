## 🔄 STRATEGIC PIVOT LOG — ทำไมถึงเปลี่ยนทิศทาง (WHY ไม่ใช่ WHAT)

### WHY: Expert migration → Source extraction
**เดิม**: ย้าย KB V2.3 (342 rules จาก Gemini) ไปยัง V2.4 schema
**ปัญหาที่เจอ**: Gemini extraction มี lossy translation — บางกฎหาย บางกฎ generalise ผิด บางกฎ copy case_study เป็น match rule
**สิ่งที่เสียถ้าใช้เดิม**: V2.4 engine ดีแค่ไหน ก็ไม่ช่วยถ้า primitive rules ผิดตั้งแต่ต้น
**การตัดสินใจ**: extract ใหม่จาก source text (docx) ทุกครั้ง — ใช้เวลาเพิ่มแต่ quality ดีกว่าแก้ legacy

### WHY: One-pass → Multi-model pipeline
**เดิม**: model เดียว extract ครั้งเดียว
**ปัญหาที่เจอ**: LLM มี hallucination — ไม่มีทางรู้ว่าผิดตรงไหนโดยไม่มีจุดเปรียบเทียบ
**สิ่งที่เสียถ้าใช้เดิม**: กฎที่ model "คิดเอง" จะปนกับกฎจริง — detect ไม่ได้ในเวลาที่เหมาะสม
**การตัดสินใจ**: 3 models extract อิสระ → compare → consensus = ground truth ที่น่าเชื่อถือกว่า

### WHY: Haiku → ต้องการ Claude-level reasoning
**เดิม**: ใช้ Haiku (เร็ว ถูก) สำหรับ extraction
**ปัญหาที่เจอ**: Principle derivation ต้องการ reasoning ข้าม N กรณีศึกษา — ไม่ใช่แค่ pattern matching
  - กรณีศึกษา 1: transit X → natal Y เมื่อ (X,Y) เป็นคู่มิตร → โชค
  - กรณีศึกษา 2,3,4: แบบเดียวกันแต่ดาวต่างกัน
  - Principle ที่ถูกต้อง: "N คู่มิตร activated พร้อมกัน → โชคใหญ่" (ไม่ใช่ copy แต่ละกรณี)
  - Haiku copy กรณีทีละอัน แต่ไม่ abstract ขึ้นมาเป็น principle
**การตัดสินใจ**: Claude (Sonnet+) สำหรับ principle derivation, Groq/Typhoon สำหรับ structured extraction

### WHY: ห้ามใช้ ANTHROPIC_API_KEY (Max Plan Only)
**เดิม**: ใช้ API key เรียก Claude โดยตรงจาก script
**ปัญหา**: policy ของ project ใช้ได้แค่ Max Plan — ห้าม key เพิ่มเติม
**ผลที่ตามมา**: Claude ทำได้แค่ in-session (interactive) ไม่ใช่ automated script
**การตัดสินใจ**: queue file pattern — Claude อ่าน queue, process batch, update queue, commit — user trigger แต่ละ session เอง

### WHY: Groq ดีกว่า Typhoon สำหรับ structured extraction
**ยังไม่พิสูจน์** — นี่คือ hypothesis ที่ต้องทดสอบด้วย m0_hallucination_test.html
**อย่าสรุปก่อนวิ่ง test**: Typhoon อาจเข้าใจ Thai astrology terminology ดีกว่าก็ได้

## 🔍 PASS/FAIL CRITERIA สำหรับแต่ละ Model

### Principle Derivation Test (CH036 — ground truth)
```
Input: บทที่มีกรณีศึกษา 5 ข้อของ transit X→natal Y คู่มิตรต่างกัน

Expected output:
  principle: "N คู่มิตร activated พร้อมกัน → โชคใหญ่ (N≥3), N=2 โชค"
  negative:  "คู่ศัตรู activated → ทุกขลาภ"
  case_study[]: 5 entries (ตัวอย่าง ไม่ใช่ match rules)

FAIL pattern:
  5 match rules ที่ copy กรณีทีละอัน (model ไม่ abstract)
  หรือ principle เพี้ยน เช่น "transit ดาวมิตร → ดี" (ขาด N คู่พร้อมกัน)
```

### Atomic Rule Test (มาตรฐาน)
```
PASS: conditions[] ตรงกับข้อความ, planet_ids ถูก, tier logic ถูก
FAIL: hallucinate planet_ids ที่ไม่มีในข้อความ, tier ผิด (1 ดาว แต่ tier 2)
```
