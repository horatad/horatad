## 🧠 ARCHITECTURE BREAKTHROUGH — อ่านก่อนทุก session

### Rule Taxonomy (3 types)
```
Type 1: Natal Atomic
        ดาว X natal คุณภาพ Y → ลักษณะ/ศักยภาพ (คงที่ตลอดชีวิต)

Type 2: Natal Combination
        ดาว X natal + ดาว Y natal → ศักยภาพซ่อนอยู่ (คงที่)

Type 3: Transit × Natal
        transit pattern → natal pattern → เหตุการณ์ "เมื่อไหร่" (เปลี่ยนตามเวลา)
```

### Tier System (color mixing — ไม่มีการลบล้าง)
```
Tier 1: planet เดียวเป็น core
Tier 2: 2 planets — emergent meaning เกิดเฉพาะเมื่อรวมกัน
Tier 3: 3 planets — emergent อีกชั้น
Tier N: recursive

กฎ: Tier ≠ จำนวน conditions[] — เป็น semantic independence level
ทดสอบ: "ถ้าเอา planet X ออก rule ยังมีความหมายไหม?" → ถ้าใช่ = tier ต่ำกว่า
```

### Intensity = ไม่ hardcode ใน rule
```
intensity = planet_quality × aspect_to_lagna × tanulak_quality × tanulak_aspect
→ engine คำนวณ runtime ไม่ใช่ extractor
```

### Lagna + Tanulak = most important
```
ลัคนา   = reference point ของทุก rule
ตนุลัคน์ = multiplier ที่ขยาย/ลด expression ทุก rule
คนเกิดวันเดียวเวลาเดียว → natal เหมือนกัน แต่ผลต่างกันเพราะ transit ต่างเวลา
```

## WHY LOG
- **Rule taxonomy 3 types** — natal atomic / natal combo / transit×natal: สำคัญมาก เป็น foundation ของ extraction ทั้งหมด. Gemini extraction พังเพราะไม่แยก type
- **Tier = color mixing** — ไม่ใช่ negation. Atomic rules ยังคง fire อยู่เสมอ composite เพิ่มความหมายใหม่ขึ้นมา ไม่ได้ยกเลิกของเดิม
- **Intensity ไม่ hardcode** — engine คำนวณจาก quality × aspect × tanulak. extractor แค่ capture rule และ meaning
- **Lagna + Tanulak = core** — ทุก prediction rule ต้องผ่าน lagna. rules ที่ไม่มี lagna reference = reference/principle ไม่ใช่ match rule
- **CH036 = case studies ไม่ใช่ rules** — กรณีที่ 1-5 เป็น illustration ของ principle "คู่มิตร N คู่". extractor ต้อง derive principle จาก pattern ไม่ copy กรณี
- **Typhoon ผ่าน CF สำหรับ browser เท่านั้น** — extraction pipeline ใช้ direct API call. key แยกกัน: CF Worker Secrets vs Claude Code env vars
