## WHY LOG

- **ลบ tag system + chips ทั้งหมด (V3.3.84, 89)** — tag ชื่อแก้บัก 6 รอบไม่หาย (handoff v2) + chips รายชื่อบนฟอร์มยังทำให้สับสนกับ tag เดิม → พี่ปีเตอร์สั่งลบทิ้งให้หมด ไม่ใช่ซ่อน. ลบ root cause ดีกว่าแก้ symptom ซ้ำๆ
- **source_id ใช้ namespace เดียวกับ JULIAN (`wikidata:Qxxx`)** — เพื่อ cross-DB lookup ตรงตัวไม่ต้อง map. public figure มี identity เดียวทุก database = dedup ได้ + track genealogy ข้าม Tank↔JULIAN ได้
- **relate_id ใช้ source_id array (bidirectional) ไม่ใช่ JD array แบบ JULIAN** — JD ชนกันได้ถ้าคนสองคนเกิดวันเดียว, source_id unique. bidirectional edge → O(1) read ต่อ hop (genealogy เร็วสุด). JULIAN's JD-based relate_id เป็น namespace แยก → JULIAN node = leaf ใน graph เรา
- **เลือก unify store (ไม่ใช่ match identity ข้าม 2 store)** — dual storage = root cause บัก tag ตลอด. มี 800 records ไม่มี user data → clean slate redesign คุ้มกว่า patch. facade strategy ลดความเสี่ยง 155 call sites
- **clear/delete memory ต้อง refresh ทุก derived view** — บัก V3.3.88: confirmClearTank/confirmClearMemory/_confirmDeleteMemory เคลียร์ Tank แต่ไม่ refresh chips บนฟอร์ม → "เหมือน memory ไม่ถูกเคลียร์". (V3.3.89 ลบ chips ทิ้งเลยจึงไม่มีปัญหานี้อีก — แต่ pattern ยังใช้ได้: state เปลี่ยน → refresh ทุก view ที่ derive จาก state นั้น)
