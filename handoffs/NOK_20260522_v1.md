## WHY LOG
- **เลือก Web Speech API ไม่ใช่ Cloud TTS** — Phase 1 MVP เน้น ฟรี + offline + 0ms latency ตรง 4 ปรัชญา (Fast > Simple > UX > Friendly) cloud TTS ค่อยมา Phase 3 เป็น fallback สำหรับ desktop ที่ไม่มี Thai voice
- **แยก project NOK ออกจาก HORATAD/PLATFORM** — TTS มี roadmap ของตัวเอง (voice profile, cloud fallback, audio export, podcast) ถ้าผูกกับ HORATAD จะ scope creep ถ้าผูกกับ PLATFORM จะ couple กับ chatbot/video ที่ยังไม่เริ่ม
- **strip emoji/markdown ก่อนพูด** — TTS อ่าน "white check mark" "asterisk asterisk" → user งง ให้ NOK side รับผิดชอบ ไม่ต้องแก้ BIBLE wording
- **chunk-split 180 ตัวอักษร** — iOS Safari มี bug pause ที่ ~15 วินาที ถ้า utterance เดียวยาวเกิน → แบ่งเป็น chunk เล็ก ๆ ให้ playback เรียงต่อกัน
- **กดซ้ำ = หยุด (toggle pattern)** — simpler กว่า 2 ปุ่ม (Play/Stop แยก) — user เห็นปุ่มเดียวเสมอ ลด cognitive load
- **ใช้ `v3/tts.js` ชื่อเดิมแทน `nok.js`** — tts.js สื่อ technical purpose ชัดกว่า ตรง pattern engine.js/typhoon.js/interpretation.js (ชื่อตามหน้าที่ ไม่ใช่ชื่อ project)
- **ปุ่มสีทอง (#b8860b) ปกติ → แดง (#dc2626) ขณะพูด** — visual feedback ชัดเจน mobile-first user มอง 2 วินาทีรู้สถานะทันที ไม่ต้องอ่าน label
- **Defer Background music (Phase 2.5)** — voice ยังไม่ test mobile ทำ BGM ก่อนคือ premature + ดนตรี subjective (user บางคนเกลียด) → wait & see model: deploy voice ก่อน รอ user request BGM แล้วค่อยทำ (data-driven) curated sample list ไว้ใน DEFERRED แล้ว user fish ดูเมื่อ ready
