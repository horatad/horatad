## WHY LOG
- **ใช้ R034 R035 R036 เป็น single source of truth สำหรับ FRIEND/ENEMY/SAMPHON** — ไม่ไปอ้างตำราโหราภายนอกเพราะอาจขัดกับสไตล์สุริยยาตร์ไทยที่ user ใช้ใน kb. หากเปลี่ยน rules 3 ข้อนี้ภายหลัง ต้อง bump engine version + audit downstream
- **helpers อยู่ใน engine.js (ไม่แยก relations.js)** — เก็บ master dictionary ไว้ที่เดียวกับ KASET_MAP/EXALT_MAP เพื่อหา reference ได้ง่าย และ engine.js comment เดิมห้ามแก้ "logic คำนวณดาว" — helper ใหม่ไม่ใช่ raw computation
- **ไม่แก้ kb.json ในรอบนี้** — schema V2.4 ต้อง user/expert approve ก่อน ถ้า migrate ก่อน lock-in ผิดจะรื้อยาก (CLAUDE.md: "Architecture lock-in ระยะยาว → ถามก่อน")
- **CASE_STUDY ติดเป็น match → noise** — แยก rule_use field แทน rule_type เพราะ rule_type เดิมใช้เป็น semantic tag (DEFINITION/HOUSE_CONCEPT) ไม่ใช่ pipeline filter
- **`global_conditions` แทน planet-level "evil_count"** — รวมในตำแหน่งเดียวต่อ rule ไม่ใช่ duplicate ต่อ planet (95 rules มี 3+ required → จะกระจาย global_conditions ทุก condition เป็น noise)
- **`predictions[]` array แทน split `p` ด้วย regex** — user/expert ระบุ semantic boundary ตอน migration ไม่ใช่ heuristic split (ที่อาจตัดผิดความหมาย "นุ่มนวล, เสน่ห์" → ควรรวมเป็นกลุ่ม)
- **DSL applies_when แทน free-form** — engine evaluable → ระบบเลือก prediction ตามสถานการณ์ user แทนแสดงทุก nuance พร้อมกัน. คู่กับ Claude infer + expert flag ทำให้ migration scale ได้
- **evil_count aspect_min = LENG strict default** — เลือก false negative มากกว่า false positive เพราะ "เครื่องเตือนเสียงดังเกินไปกลายเป็นไม่ได้ยิน" (CLAUDE.md philosophy: "ไม่ขัดจังหวะ workflow")
- **engine_min_version refuse (ไม่ใช่ warning)** — silent corruption เลวกว่า dead-end + auto-reload เพราะ trust ของ user หายแก้กลับยาก
- **LAR_NORMALIZE map ไทย→ENG** — kb V2.3 ใช้ 'เล็งลัคนา' แต่ engine คำนวณ 'LENG' → bug เดิมที่ existing code ใช้ string compare ผิด. ตอนนี้ normalize ทุกครั้งใน _checkCondition → V2.3 rules ที่เคย fail silently จะเริ่ม match ถูกต้อง — **ต้องทดสอบว่าจำนวน matched rules ใน V3 tab ไม่เปลี่ยนแบบรุนแรง** (28 → 28 ใน test session นี้ OK)
