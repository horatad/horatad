## WHY LOG

- **เลือก 100K ไม่ใช่ 200K ตามที่ user ขอ "ขยาย"** — sweet spot analysis ชี้ชัด: 100K ได้ value/time ดีที่สุด, 200K marginal value ตก threshold (V/T ratio ดรอป 10x)
  → user สัมผัสได้: ครบเร็วกว่า, GitHub Actions usage ต่ำกว่า, BIBLE validation ยัง excellent
  → ถ้าต้องการเพิ่ม → ทำ Phase 3 (100K → 200K) ทีหลังหลัง BIBLE ผลออก

- **MILESTONES = [10K, 25K, 50K, 75K, 100K] ไม่ใช่ทุก 10K** — ไม่อยากสร้าง backup branch ทุก run (รก) แต่ก็ไม่อยากห่างเกินไป (กลัวข้อมูลหายระหว่าง milestone)
  → 5 milestones ที่ business meaningful (statistical thresholds)

- **Sanity check threshold = 5%** — เผื่อ legitimate drop ที่อาจเกิด (audit cleanup, manual fix) แต่ block accidental wipe
  → ถ้า workflow abort เพราะ sanity → user รู้ทันที (commit ไม่ขึ้น) ไม่ใช่ silent fail

- **Backup branch ไม่ใช่ git tag** — sandbox CLI environment เคยเจอ tag push 403, branch push ผ่าน → ใช้ pattern เดิมที่ทำงานเสมอ

- **3 category queries ใหม่ทำตอนนี้ ไม่ต้องรอรอบ 2** — capacity เพียงพอใน config, sitelinks filter (>= 15) คุมคุณภาพแล้ว, รวมกัน expected +15K records (ของจริงน้อยกว่าเพราะ dedup กับ era queries)
