## 📚 HOSTING ARCHITECTURE REFERENCE

### Pattern A — GitHub Pages + Cloudflare proxy (current horatad) ⭐
```
DNS (Cloudflare) → CNAME → GitHub Pages → static files
                ↓
            Workers (auth, AI logic)
```
- ฟรี ทุกอย่าง
- App fetch data ผ่าน `raw.githubusercontent.com` (repo ต้อง public)
- Security ได้จาก Cloudflare proxy (DDoS, WAF, rate limit, TLS)

### Pattern B — Cloudflare Pages + Workers (alternative)
```
CF Pages (deploy from repo) ─ horatad.com
       ↓
   Workers + R2 + KV
```
- ฟรี ทุกอย่าง
- Repo private ได้ (CF Pages รองรับ)
- Preview deployments per branch

### เมื่อไหร่ย้าย Pattern A → B

ย้ายเมื่อ trigger เหล่านี้:

- GitHub Pages bandwidth quota เริ่มเต็ม (100 GB/mo)

## WHY LOG

### Security decisions

- **Repo stays public — moat = domain expertise, not code visibility (2026-05-25)** — ตัดสินใจไม่เปลี่ยน private เพราะ (1) ต้นทุน sustained (app พัง + $48/yr Pro หรือ migrate hosting), (2) cosmetic block (fork ปิดไม่ได้บน personal repo, clone ทำได้อยู่ดี) ไม่ป้องกันจริง. แทน: ปิด channel ทั้งหมด + README license + contact ผ่าน horatad.com เท่านั้น. หลักการ: **"หัวใจอยู่ที่ engine คำนวณสุริยยาตร์ + KB rules ที่สะสมจากหนังสือ — niche knowledge ที่คน copy code ไม่สามารถ replicate ได้ทันที"**

- **2FA = highest-leverage security action (2026-05-25)** — account-level setting, 1 ครั้งครอบคลุมทุก repo current + future ตลอดไป. ทำก่อนทุก security setting อื่น เพราะป้องกัน root cause (account compromise = repo ถูกลบ/แก้ทั้งหมด).

- **Branch protection ruleset ลบทิ้ง (2026-05-25)** — สร้างแล้วทดสอบ → sandbox token ไม่ถูก recognize เป็น Repository admin → push main ติด 403 ทุกครั้ง → toggle disable ทุก session = friction สูง. แทน: รอ defense ผ่าน (1) 2FA, (2) CLAUDE.md rule "ห้าม force-push", (3) backup/* branches per version, (4) git history immutable หลัง push (ถ้าไม่มี force-push). Scenario ที่ ruleset กันได้ = แทบไม่เกิด.

- **Cloudflare = main security layer สำหรับ horatad.com (2026-05-25)** — DDoS / WAF / rate limit / TLS / bot management อยู่ที่ CF proxy. docs/cia/ public ลด impact ลงเพราะ network attack กัน CF ได้แล้ว. Code-level security (XSS, logic bug) เป็นเรื่องของ source ไม่ใช่ hosting → patch ใน code (P1-D ทำแล้ว).

### Hosting decisions

- **Pattern A (GitHub Pages + CF proxy) ปัจจุบัน = optimal** — ฟรี ทุกอย่าง + ครบ feature ที่ใช้. ไม่ย้ายเป็น CF Pages ตอนนี้เพราะ migration effort > benefit. Trigger ย้าย: private repo / CF stack consolidation / preview branches needed.

- **Lesson learned: GitHub raw URL pattern lock-in repo public** — ตัดสินใจเดิม "GitHub raw แทน Cloudflare Release" (CORS-free, no token expire) ลึก ๆ บังคับ repo เป็น public ตลอด. ถ้าจะย้าย private = ต้องแก้ data fetch architecture พร้อมกัน.

### Process decisions

- **ตัด local main branch cleanup ออกจาก Claude scope** — sandbox 403 delete remote branch → ทำไม่ได้ ปล่อย user
- **BIG admin role decision policy** — เมื่อ user มอบ admin: ทำงาน sync/doc/handoff (low risk, reversible) ทันที, ข้ามงานที่มี hard dependency บน external action (user / repo create / sandbox limitation)
- **Cross-repo restriction (sandbox)** — sandbox restricted ไป `horatad/horatad` เท่านั้น → ไม่สามารถสร้าง/push repo อื่น (เช่น finger-scan) → user ต้องทำผ่าน GitHub UI เอง

### Finger-scan handling

- **ลบ fingerprint files หลัง user create new repo** — เดิมตั้งใจรอ user ย้ายเสร็จก่อนลบ. ตัดสินใจลบทันที (44512c1) เพราะ git history ยัง recover ได้ที่ SHA `74a1e7d`. User สามารถดึงไฟล์ผ่าน raw URL ของ historic SHA + upload UI เข้า repo ใหม่ — ทำได้แม้ไม่มี local clone.
- **finger-scan = private (ฟรี)** — desktop hardware project ไม่ต้องการ GitHub Pages / raw URL → private ได้โดยไม่กระทบ + ไม่เสียเงิน
